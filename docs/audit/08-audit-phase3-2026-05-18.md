## 二、发现的问题（Problems）

### P-01 `backtest_page.py` 从 260 行暴增至 797 行（严重性：中）

**现象：** 将 12 个 sweep helper 函数迁入 `backtest_page.py` 后，该文件从约 260 行增至 797 行，内部函数数量达 14 个。

**根因：** 迁移策略是"把 gui.py 的私有函数移进 backtest_page.py 作为模块级私有函数"。在解决 gui.py 边界问题的同时，制造了一个新的单文件过大问题。

**影响：** `backtest_page.py` 现在同时承载：
- 回测主渲染流程（`render_backtest_page`）
- 参数扫描渲染流程（`_render_parameter_debug_section`）
- `@st.cache_data` 计算缓存（`_cached_parameter_sweep`）
- 数据本地化逻辑（`_localized_*`、`_with_parameter_ui_names`）
- 图表构建逻辑（`_sweep_comparison_curves`、`_sweep_factor_curves`）
- 图表渲染逻辑（`_sweep_metric_line_chart`）

六类不同职责混在一个文件里，与 gui.py 的历史问题结构完全一致。

### P-03 `_cached_parameter_sweep` 的哈希不稳定风险（严重性：中）

**现象：** 函数现在位于 `backtest_page.py:500`，接受 `pd.Series`（price、vix）和 `dict`（model_settings，包含嵌套 VIX 规则列表）作为参数。

**风险：** Streamlit Issue [#10957](https://github.com/streamlit/streamlit/issues/10957) 确认：`dict` 中含 `list` 时，`@st.cache_data` 的哈希可能不稳定，导致每次 rerun 都触发重新计算，缓存失效。

**当前缓解：** 无。参数未添加 `_` 前缀，未使用 `hash_funcs`。

**已知可行方案（见第五节）：**
1. 使用 `_` 前缀跳过哈希（危险：完全绕过缓存逻辑）
2. 提取稳定的基本类型参数传入（推荐）
3. 自定义 `hash_funcs={dict: lambda d: json.dumps(d, sort_keys=True, default=str)}`

### P-05 `--leo-metallic-green` 偏离真实英国赛车绿（严重性：低，设计正确性）

**现象：** 当前 token：
```css
--leo-metallic-green: rgba(18, 78, 60, 0.88);   /* = teal-green，偏蓝绿 */
```

**官方英国赛车绿（British Racing Green）：**
- 最常见标准：`#004225` = `rgb(0, 66, 37)` — 深暗、近橄榄、不带蓝色分量
- 变体：`#033C28` = `rgb(3, 60, 40)` / `#004B2B` = `rgb(0, 75, 43)`

当前 `rgb(18, 78, 60)` 的蓝色分量（60）过高，绿色分量（78）偏浅，整体色相偏青绿（teal），不符合正宗 BRG 的深橄榄绿色调。

**参考对比：**

| 名称 | RGB | 色相描述 |
|------|-----|---------|
| 官方 BRG（最常见）| rgb(0, 66, 37) | 深暗橄榄绿，无蓝调 |
| 变体 BRG | rgb(3, 60, 40) | 更深，接近黑绿 |
| 当前 token | rgb(18, 78, 60) | 偏青绿/蓝绿（teal） |
| 既有 `--leo-racing-green` | rgba(31, 106, 83, 0.88) | 明亮赛车绿，用于 UI accent |

来源：[colorhexa.com/#004225](https://www.colorhexa.com/004225)，[color-register.org](https://color-register.org/color/british-racing-green)

## 三、发现的风险（Risks）

### R-01 `backtest_page.py` 下一步提取的边界判断难度增大

**分析：** 797 行文件内混合了计算缓存、本地化、渲染和 PDF 生成四类逻辑。如果继续不加分层地往里加代码，两个月后 `backtest_page.py` 就会成为下一个 gui.py。

**判断：** 当前规模尚可接受；但需要在进入 Phase 4 之前建立明确的分层边界，避免路径依赖。

---

### R-02 `@st.cache_data` 与 `AppTest` 跨测试持久化（测试污染）

**分析：** Streamlit 官方已确认（Issue [#9139](https://github.com/streamlit/streamlit/issues/9139)）：`@st.cache_data` 的缓存在同一 Python 进程中的多个 `AppTest` 实例之间持久化。当前测试全为静态断言，不受影响；但若未来引入 `AppTest` 集成测试，缓存污染会导致测试顺序依赖和偶发失败。

**缓解：** 在集成测试中调用 `st.cache_data.clear()` 作为 fixture teardown，或使用 pytest 的进程隔离模式。

---

### R-03 `HEADER_UI_THEME` 可静默覆盖 `SETTINGS_UI_THEME`（无文档）

**现象（`gui.py:_apply_session_preferences`）：**
```python
if SessionKeys.SETTINGS_UI_THEME in st.session_state:
    st.session_state[SessionKeys.UI_THEME] = st.session_state[SessionKeys.SETTINGS_UI_THEME]
if SessionKeys.HEADER_UI_THEME in st.session_state:
    st.session_state[SessionKeys.UI_THEME] = st.session_state[SessionKeys.HEADER_UI_THEME]
```

如果两个键同时存在，`HEADER_UI_THEME` 会静默覆盖 `SETTINGS_UI_THEME`，用户在 Settings 页面设置的主题偏好会被 header 控件覆盖。这一优先级顺序没有文档记录，也没有测试覆盖。

**风险：** 用户无法通过 Settings 页面持久设置主题，只能靠 header 控件；这可能不是预期行为。

---

### R-04 `default_raw_settings=lambda: load_settings(DEFAULT_CONFIG).raw` 硬编码依赖

**现象：** `BacktestPageDeps` 仍接收该 lambda，并在扫描时调用以取默认配置（用于基准对比）。若 `DEFAULT_CONFIG` 文件不存在或格式变化，sweeper 会静默失败或抛出异常。

**风险：** 该 lambda 不可测试（无法在 `AppTest` 中 mock），且错误不会在 `py_compile` 阶段被捕获。

---

### R-05 截图基线无自动对比机制

**现象：** 8 张 PNG 存在于 `docs/audit/screenshots/`，但没有任何代码或 CI 脚本将新截图与基线对比。

**影响：** 基线成为"存档参考"而非"回归保护网"。UI 变化后不会自动告警。

### I-02 修复 `_cached_parameter_sweep` 哈希稳定性

**方案（推荐，最小改动）：**
```python
import json

@st.cache_data(ttl=3600)
def _cached_parameter_sweep(
    price: pd.Series,
    vix: pd.Series,
    model_settings: dict,
    ...
):
    ...
```
改为：
```python
@st.cache_data(ttl=3600, hash_funcs={dict: lambda d: json.dumps(d, sort_keys=True, default=str)})
def _cached_parameter_sweep(...):
    ...
```

这让 dict（含嵌套 list）通过 JSON 序列化取得稳定哈希，避免每次 rerun 重算。

**备选方案（最简单，有风险）：** 将 `model_settings` 参数名改为 `_model_settings`，Streamlit 会跳过对该参数的哈希，永远使用第一次调用的缓存。仅在确认 model_settings 不影响输出正确性时可用。

## 七、白天模式失效根因专项排查（2026-05-18 增补）

### 7.1 现象描述

切换到白天模式后，**只有部分元素变白**，其余文字、图标、metric 标签、卡片描边仍然保留夜间状态的奶油白／低饱和度配色。表现为"半亮半暗"的破碎主题，而非彻底的浅色配色。

### 7.2 根因定位 — 一段假装存在的 DOM 钩子

**结论：这不是 Streamlit 主题选择器与页面引擎"冲突"，而是页面主题系统从一开始就建立在一个不存在的 DOM 属性上。**

#### 证据链

**① 全部主题样式都挂载在 `[data-theme="X"]` 选择器上。** 在 `base.css`、`components.css`、`shell.css`、`preparing.css` 中共计 **34 条** `[data-theme="dark"]` 复合选择器规则，覆盖 stMetric、stMetricValue、stMetricLabel、stMetricDelta、stHeading、sidebar 标签、slider、stStatusWidget、role=switch、app-shell-nav 等几乎所有受影响组件。

**② 整个工程从未将 `data-theme` 属性写入 DOM。**
```bash
$ grep -rn "setAttribute\|data-theme" trend_system/ --include="*.py"
shared/theme.py:43:    selector = f'[data-theme="{theme}"]'   # 仅用于 CSS 提取，不写 DOM
shared/tradingview_chart.py:100: parentDoc?.documentElement?.getAttribute("data-theme")
shared/tradingview_chart.py:108: document.body.setAttribute("data-theme", parentTheme)  # 仅 iframe 内部
```
**外层文档的 `<html>` / `<body>` / `.stApp` 上从来没有 `data-theme="light"` 或 `data-theme="dark"` 属性。** TradingView iframe 内部那两行只针对图表 iframe 自身的 `body`。

**③ `theme_override_text(theme)` 用正则做的"伪 DOM 切换"是不对称的。**
```python
# shared/theme.py:43–49
selector = f'[data-theme="{theme}"]'
blocks   = re.findall(rf"{re.escape(selector)}[^\{{]*\{{[^{{}}]*\}}", ...)
for block in blocks:
    normalized.append(block.replace(selector, ":root"))
```
- 对于 **token 重定义块**（`[data-theme="dark"] { --leo-page-bg: #1A1D1F; }`）：正则替换为 `:root { --leo-page-bg: #1A1D1F; }` → ✅ token 翻转生效。
- 对于 **复合选择器块**（`[data-theme="dark"] [data-testid="stMetric"] { background: ... }`）：正则同样匹配，替换为 `:root [data-testid="stMetric"] { ... }` → 这等价于"任何 `:root` 后代的 stMetric"，**永远命中** —— 与选择的主题无关。
- 对于 **light 模式**：只有 `[data-theme="light"]` 块被提取。原始 CSS 中的 34 条 `[data-theme="dark"] ...` 复合规则继续躺在样式表里，因为 DOM 上没有 `data-theme` 属性，它们一条都不匹配 → 夜间专属配色应该完全不渲染。
- 对于 **dark 模式**：dark token 块被提取并提升到 `:root`，dark 复合规则也被改写为 `:root ...` 后被注入 → 夜间配色完整生效。

**所以在 light 模式下应当看到一个干净的 light 渲染才对，为什么没看到？**

#### 真正的故障点 — Light 模式覆盖块严重缺失

继续核查 `tokens.css` 与各样式文件的 `[data-theme="light"]` 块：

```bash
$ grep -c "^\[data-theme=\"light\"\]\|^\[data-theme=\"dark\"\]" styles/*.css
tokens.css         → 1 light块 / 1 dark块
base.css           → 0 light块 / 7 dark复合块（含一条 dark-only metric override）
components.css     → 0 light块 / 13 dark复合块（metric, heading, slider, statusWidget...）
shell.css          → 0 light块 / 4 dark复合块（app-shell-nav 三态）
preparing.css      → 0 light块 / 1 dark复合块
```

整套 CSS **没有一条 `[data-theme="light"] ...` 的组件级覆盖规则**。Light 模式的全部"翻转"动作完全依赖 `tokens.css:39–58` 那一个 token 块。

但许多组件规则在**默认（无主题前缀）** 块里就硬编码了"看起来应该走 light"的颜色，比如：
- `components.css:55` 默认 `.leo-section-head--prussian .leo-section-overline { color: rgba(244, 240, 232, 0.94) !important; }` — **奶油白文字直接写死在默认块里**
- `components.css:201` 默认 `color: rgba(244, 240, 232, 0.90);` — 同样写死奶油白
- `components.css:336` 默认 `color: rgba(244, 240, 232, 0.92) !important;` — 同样
- `components.css:311` 默认 `background: linear-gradient(..., rgba(26,29,31,0.55) 100%) !important;` — 深色背景直接写死

这些规则**没有任何 `[data-theme="dark"]` 前缀**，意味着它们在 light 和 dark 模式下都生效。Light 模式下 token 翻转把"页面底色、ink、surface"变浅了，但这些**硬编码的奶油白文字 / 深色背景照旧渲染** —— 这就是用户看到的"部分变白、其他仍然是黑"的真实成因。

### 7.3 与 Streamlit 引擎的关系 — 不是冲突，是错认

经查 Streamlit 1.27+ 源码与社区调研（GitHub issue [#6456](https://github.com/streamlit/streamlit/issues/6456)、[#5437](https://github.com/streamlit/streamlit/issues/5437)、[#8370](https://github.com/streamlit/streamlit/issues/8370)、[#10660](https://github.com/streamlit/streamlit/issues/10660)）：

| 假设 | 是否成立 | 说明 |
|------|---------|------|
| Streamlit 在 `<body>` 上写 `data-theme="X"` | ❌ 否 | Streamlit 通过 styled-components / emotion 注入内联 class，不暴露 DOM 主题钩子。社区开 issue 长期请求该属性，至今未实现 |
| Streamlit 主题选择器与本工程页面架构冲突 | ❌ 否 | 没有冲突点。`st.set_page_config` 不操作 CSS 变量，本工程注入的 `<style>` 也不被 Streamlit 引擎清理 |
| 本工程把 `data-theme` 当成"Streamlit 应该会自动加的属性" | ✅ **是** | 这是核心误解。整套样式系统建立在一个永远不会出现的 DOM 属性上，靠 `theme_override_text` 的正则做"假装翻转"来部分填补，但只填到 token 这一层，组件级覆盖完全没有 light 对应规则 |

**用户问题中的假设 (a) 不成立。** 真正的原因是假设 (b)：页面主题系统的设计前提（"Streamlit / 浏览器会自己设 data-theme"）从未落地，是工程内部的架构缺陷，与 Streamlit 引擎无关。

### 7.4 修复方案（按工程量从小到大）

#### 方案 A — 永久锁定夜间模式（推荐，最低成本）

**理由：**
- 整套 CSS 已经为夜间模式打磨了 34 条复合规则、金属边框、奶油白文字、暗 BRG 配色；light 模式从设计上就是不完整的
- 工程目标是为 Leo 个人使用的金融分析终端，长期视觉风格已收敛到"暗色仪表盘 + 金属感配饰"
- 切换器只会引入持续的视觉一致性维护负担

**实施步骤：**

1. **创建 `.streamlit/config.toml`：**
   ```toml
   [theme]
   base = "dark"
   primaryColor       = "#12395B"
   backgroundColor    = "#1A1D1F"
   secondaryBackgroundColor = "#141618"
   textColor          = "#F4F0E8"
   font               = "sans-serif"

   # 让用户即使在 Settings 里切到 light 也仍然得到同一套暗色：
   [theme.light]
   base = "dark"
   backgroundColor    = "#1A1D1F"
   textColor          = "#F4F0E8"

   [theme.dark]
   base = "dark"
   backgroundColor    = "#1A1D1F"
   textColor          = "#F4F0E8"
   ```

2. **代码侧收口：**
   - `shared/theme.py:resolve_theme()` 直接 `return "dark"`，忽略 settings 与 session
   - `settings_page.py` 删除主题 selectbox，或保留为只读 disabled 状态附说明
   - `_render_shell_header` 删除右上角的主题切换按钮（如有）
   - `theme_override_text()` 改为只注入 dark 块（或彻底删除该函数，把 dark token 直接写进 `:root`）

3. **CSS 简化（可选，二阶段）：**
   - `tokens.css` 删除 `[data-theme="light"]` 与 `[data-theme="dark"]` 两个块，把 dark 值合并进 `:root`
   - 34 条 `[data-theme="dark"] ...` 复合规则去掉前缀，直接以默认规则形式存在
   - 删除整个 `theme_override_text()` 与相关代码路径

**收益：**
- 彻底消除"半亮半暗"现象
- CSS 体积可缩减约 15–20%（去掉重复的 light 块与 dark 前缀）
- 维护面缩小 50%（不再需要为每个新组件写两套配色）

**风险：** 不可逆变更。如果未来确实需要 light 模式，需要补一整套 light 复合规则，工作量约 1–2 天。

---

#### 方案 B — 真正实现 light 模式（高成本，不推荐）

如果坚持保留双主题：

1. **写入 `data-theme` 到 DOM：** 在 `inject_styles(theme)` 内追加 JS：
   ```python
   st.markdown(
       f"""
       <script>
         (function(){{
           const root = window.parent.document.documentElement;
           root.setAttribute('data-theme', '{theme}');
         }})();
       </script>
       """,
       unsafe_allow_html=True,
   )
   ```
   注意：`st.markdown(unsafe_allow_html=True)` 默认会清理 `<script>`，需要改用 `streamlit.components.v1.html` 或 `streamlit-javascript` 组件执行。
2. **补齐 light 复合规则：** 为现有的 34 条 `[data-theme="dark"] ...` 中每一条补对应的 `[data-theme="light"] ...`，统一覆盖 ink、background、border、shadow。
3. **审计所有硬编码颜色：** `components.css:55/201/311/336` 等位置的硬编码奶油白／深背景必须挪进 `[data-theme="dark"]` 前缀块；默认块改用 token。

**工作量估计：** 2–3 天 CSS 编写 + 1 天浏览器 QA + 截图基线重做。

---

#### 方案 C — 改用 `streamlit-theme` 组件读取实际主题（中等成本）

利用社区方案 [`gabrieltempass/streamlit-theme`](https://github.com/gabrieltempass/streamlit-theme)（pip 包名 `st-theme`），在 Python 侧拿到 Streamlit 实际渲染的主题，再决定要注入哪一套 CSS。

**优点：** 不再依赖伪 `data-theme` 属性，可同时实现自动跟随 OS。  
**缺点：** 引入一个 iframe 组件依赖、首屏有短暂闪烁（与 issue #6456 描述一致）、仍然需要补齐 light 模式的全部覆盖规则。

### 7.6 推荐结论

**采用方案 A，永久锁定夜间模式。** 理由：

1. 工程已有的 CSS 资本 100% 投入在 dark 模式上，light 模式从未真正存在过 —— 只是 token 翻转了，组件覆盖完全缺失
2. 用户的实际使用场景（个人金融终端、长时间盯盘）本身就更适配深色
3. 方案 B 的工作量（≥3 天）与收益不成比例，方案 C 的 iframe 依赖会拖慢首屏
4. 锁定后可同步删除 `theme_override_text()`、`SessionKeys.UI_THEME`、settings 页主题 selectbox、F-01 三路主题写入收敛问题 —— 顺势完成多项审计待办

**实施前确认事项：** 是否保留 settings 页"主题"控件作为占位（带"已锁定为夜间模式"说明文案），还是彻底移除该项？建议**保留控件但置灰**，作为产品说明而非功能开关。
