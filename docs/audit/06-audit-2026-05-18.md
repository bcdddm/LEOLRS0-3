### §2.5 — CSS 加载改用 mtime 感知缓存 ✅ 已核实

**声明：** 移除 `lru_cache`，改用模块级 dict 按文件修改时间缓存。

**核实（theme.py）：**
```python
_css_cache: dict[str, tuple[float, str]] = {}

def stylesheet_text() -> str:
    for name in STYLE_FILES:
        path = STYLES_DIR / name
        mtime = path.stat().st_mtime
        cached = _css_cache.get(name)
        if cached is None or cached[0] != mtime:
            _css_cache[name] = (mtime, path.read_text(encoding="utf-8"))
        parts.append(_css_cache[name][1])
```

行为：CSS 文件未修改时不重读磁盘；文件 mtime 变化时自动失效重载，无需重启进程。

**已知边界条件（已记录，可接受）：** `_css_cache` 是模块级变量；在多 worker 或多进程 Streamlit 部署下每个进程持有独立缓存，不存在跨进程共享问题。

## 二、独立发现的结构问题

### F-01 主题写入有三条收敛路径（风险：低，维护成本：中）

**现象：** `UI_THEME` 在同一个 Streamlit rerun 周期内被写入三次：

```
1. settings_page.py:86        st.session_state[UI_THEME] = selected_theme
                               → st.rerun()
                               ——— 以下为 rerun 期间 ———
2. gui.py:_apply_session_preferences():136–137
                               st.session_state[UI_THEME] = st.session_state[SETTINGS_UI_THEME]
3. gui.py:main():176
                               st.session_state[UI_THEME] = resolved_theme
```

三条路径最终收敛到相同值（因为 `SETTINGS_UI_THEME` 在 rerun 后保持新值），所以功能正确。但 `settings_page.py:86` 的直接写入是多余的 — 只需 `st.rerun()` 就够，`_apply_session_preferences` 会在下一轮完成拷贝。

**风险：** 日后若 `SETTINGS_UI_THEME` 的生命周期行为改变（如 widget 被销毁或 key 被重命名），`settings_page.py:86` 的直接写入会成为唯一有效路径，而这条路径在文档中没有被明确声明为"规范写入点"。

**建议：** 将 `settings_page.py:86` 的直接 `UI_THEME` 写入删除，仅保留 `st.rerun()`，由 `_apply_session_preferences` 统一负责主题同步。
